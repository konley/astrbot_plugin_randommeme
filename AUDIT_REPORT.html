🐛 问题 (Bugs)

1. send_as_image_component 配置项从未被读取

`_conf_schema.json:20` → `main.py:203-211`
死配置 _conf_schema.json 定义了 send_as_image_component（默认 true），但 main.py 中从未调用 self.conf.get("send_as_image_component")。_build_image_chain() 总是返回 Image Component，用户即使把这个配置改成 false，行为也不会有任何变化。
修复方案（二选一）：
A) 实现逻辑：在 _build_image_chain 中读取配置，为 false 时改为文本提示路径
B) 删除配置：从 _conf_schema.json 删除此项，避免用户困惑
推荐方案 B — 该功能没有实际使用场景，文本提示路径对用户毫无意义。

2. MemeManager.reload_config() 从未被调用

`core/manager.py:52-56`
死代码 reload_config() 方法旨在运行时更新 gif_support 和 exact_match 而不重建 manager 实例。然而 main.py 的 __init__ 在每次 reload 时直接新建 MemeManager 实例，没有任何调用链能到达此方法。
修复方案：删除 reload_config() 方法（约 6 行代码），或在 RandomMemePlugin 上加一个 on_config_updated 钩子并调用之。推荐前者，因为新建实例等价且更简洁。

🧹 代码异味

_should_handle / _is_woken 存在重复逻辑

`main.py:183-201`
可重构 两个方法几乎相同 —— 都检查 extra_prefix 和 is_at_or_wake_command。实际上 _should_handle 的 true 条件包含了 _is_woken 的 true 条件，存在逻辑重叠与混淆。
建议：将 _should_handle 改为纯"是否应该处理此消息"的判断，_is_woken 改为纯"当前消息是否被唤醒"的判断，消除重叠。
或直接在 on_message 内联判断，减少间接层数。

_api_get_stats 访问私有属性 _history

`core/api.py:387,392`
封装弱化 使用 self.manager._history（# noqa: SLF001）而非公有 API，破坏封装。如果未来修改 _history 的内部表示，API handler 会静默出错。
建议：在 MemeManager 上添加公有方法 get_history_size() 和 get_group_drew_count(name)，供 API 层调用。

💡 改进建议

1. 🏆 权重抽取系统

允许为每张图片设置权重（正整数），权重越高被抽中的概率越大。打破当前"纯随机"的限制，让热门图更容易出现。
难度：中等 · 需改造 draw() 算法（加权随机 + 权重持久化）

2. ⏰ 定时发送 / 每日推送

在指定时间（如每天 10:00）自动从指定组别抽一张图发送到指定群聊。适合"每日摸鱼图"、"早安图"等场景。
难度：中等 · 需 asyncio.create_task 常驻定时器 + 群配置持久化

3. ⏱ 触发冷却 (CD)

为每个群/每个用户设置触发冷却时间（如 30 秒内只能抽一次），防止刷屏。
难度：低 · 内存 dict + timestamp 即可，注意在重启后重置

4. 📦 导出/导入

一键导出组别（含图片和配置）为 zip 包，另一侧导入。方便迁移和分享表情包合集。
难度：中等 · 后端生成 zip 流 + 前端下载/上传

5. 💾 收藏图片 / 从消息加图

用户在群中看到一张图，发 收藏表情 组别名 指令，机器人把图中的图片保存到对应组别。用户生成的 UGC 机制。
难度：中等 · 需从消息对象中提取 Image 组件 + 下载 + 去重

6. 📊 抽取统计持久化

记录每张图的被抽次数、按周/月生成热门排行。当前统计仅基于 history（洗牌即重置），无法计算长期趋势。
难度：低 · groups.json 加 draw_counts 字段 + 累加器

7. 🚫 群黑名单

允许对特定组别设置禁用的群聊列表，某些群不发某些表情。
难度：低 · Group 加 banned_groups: list[str]

8. 📝 文字随图回复

抽图时可以附带一段自定义文本（如"今天也要加油哦！"），在 Group 上加 reply_text 字段。
难度：低 · Group 加字段 + 前端表单 + 拼链

9. 🔎 前端搜索/过滤

组别管理页增加搜索框，图片较多时按文件名搜索。当前组别多时不好找。
难度：低 · 纯前端 JS 过滤

10. 🌐 Telegram 等更多适配器支持

metadata.yaml 中 support_platforms 仅声明了 aiocqhttp 和 qq_official。核心逻辑不依赖适配器特性（纯图片组件），理论上 Telegram、Discord 等也能用。
难度：低 · metadata.yaml 加几个适配器键即可

11. 🎨 图片懒加载优化

前端 makeImageCard 中每个图片都独立调用 loadImageData()，图片多时会有大量并发 API 请求。可批量接口打包请求，或前端分页懒加载。
难度：低～中 · 前端加 IntersectionObserver 懒加载

12. 📋 全局设置 Tab 增加动态性

当前 Settings Tab 是纯静态 HTML 展示。可改为从后端拉取实时配置值（如当前 gif_support、exact_match），让运维人员一目了然。
难度：低 · 后端加 /config 端点 + 前端渲染
